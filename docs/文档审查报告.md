> 审查范围：`docs/` 目录（354 个 Markdown 文件，12 个模块）
> 审查时间：2026-06-28
> 审查重点：结构规范性、命名一致性、链接有效性、内容完整性、文档元数据统一性

| 级别 | 含义 |

## 一、发现的问题清单

### 🔴 P0-1：大量死链——README 指向不存在的文档文件

多个模块的 README 列出了 00–07 全套文档链接，但实际只有部分文件存在，点击即 404。

| 模块 | README 链接的文件 | 实际缺失的文件 |
|------|-------------------|----------------|
| `cpp/modules/http2/` | 00–07 全套 | **00、01、02、03、04、06、07**（仅 05 存在） |
| `cpp/modules/ws/` | 00–07 全套 | **00–07 全部缺失**（整个模块只有 README） |

**影响**：读者按 README 导航点击链接全部落空，严重损害文档可信度。

**建议**：二选一——
1. 若模块尚未完成文档，README 中应删除未实现文件的链接，改写为"当前仅提供性能测试文档（05-性能测试.md）"；
2. 若文档是计划中的，把链接移到单独的"规划中的文档"小节并标注"待补充"。

---

### 🔴 P0-2：mongo 模块文档内部链接编号系统性错乱

`mongo` 模块的文档文件本身是标准 00–07 编号，但文档正文中的交叉引用链接编号整体偏移了，指向了错误的文件。

| 出现位置 | 错误链接 | 应为 |
|----------|----------|------|
| `00-快速开始.md:287` | `[示例代码](06-示例代码.md)` | `04-示例代码.md` |
| `00-快速开始.md:289` | `[高级主题](07-高级主题.md)` | `06-高级主题.md` |
| `07-常见问题.md:736` | `[性能测试](04-性能测试.md)` | `05-性能测试.md` |

**根因**：疑似从其他模块复制时未修正"下一步"链接的编号。

### 🔴 P0-4：ws 模块几乎无文档

`cpp/modules/ws/`（WebSocket 模块）除了一个列出全套死链的 README 外，**没有任何实质文档**——没有快速开始、没有 API 参考、没有示例代码。这与同目录下其他模块（http、redis、rpc 等都有完整 00–07）形成巨大落差。

---

### 🔴 P0-5：tracing 模块编号体系不统一

tracing 模块采用了完全不同的编号方案和文件命名，与其他所有模块不一致：

| 对比项 | 其他模块 | tracing 模块 |
|--------|----------|--------------|
| 快速开始 | `00-快速开始.md` | `01-快速开始.md` |
| 架构设计 | `01-架构设计.md` | `02-链路上下文.md` |
| API 参考 | `02-API参考.md` | 无（拆分到其他页） |
| 文件数量 | 8 个标准页 | 6 个，内容主题不同 |
| README | 有 | **无** |

tracing 完全是一套独立的文档体系，读者跨模块查阅时会困惑。

### 🟡 P1-1：模块 README 一级标题（H1）不统一

11 个有 README 的模块，H1 标题用了 5 种不同写法：

| H1 标题 | 使用模块 |
|---------|----------|
| `# 文档导航` | kernel、http、http2、ws、c/kernel |
| `# 文档索引` | redis、mysql、etcd |
| `# 文档总览` | ssl、mcp |
| `# Galay-RPC 文档总览` | rpc |
| `# 文档首页` | utils |
| **（无 README）** | mongo |

应统一为一种（推荐 `# 文档导航` 或 `# {模块名} 文档`）。

### 🟡 P1-4：docs 顶层缺少索引入口

`docs/` 目录下只有各模块子目录和零散的 `release_note.md`、`plans/`，**没有顶层 README 或 index**。新读者进入 `docs/` 后无从得知有哪些模块、应该从哪开始。

建议新增 `docs/README.md` 作为总入口，列出所有模块及一句话说明。

### 🟡 P1-5：rpc 模块多出非标准文件

`cpp/modules/rpc/` 除了标准 00–07 外，还有一个 `performance-comparison.md`，没有编号前缀，游离在标准体系外。建议归入 05-性能测试 或加上编号前缀。

### 🟢 P2-1：性能测试文档的"尚未测量"声明措辞不一

部分模块的性能测试页含"尚未在本次整改中重新测量"声明（如 mcp），但措辞和位置不统一。建议统一为一个固定的"测量状态说明"小节模板。

`.wroktree/c-coro-runtime/`（疑似 `worktree` 拼写错误）占 **1.0 GB**，内含 docs 的旧副本。虽已被 `.gitignore` 忽略（不污染仓库），但本地占用大量空间，且其中的旧文档可能造成混淆。建议清理。

## 四、做得好的地方

为客观起见，也记录当前文档的亮点：

- **kernel 模块文档体系完善**：不仅有标准 00–07，还扩展到 08–20 共 21 个专题页，且 README 中有清晰的"主题回流矩阵"和"按关键词进入"索引，是全项目文档的标杆。
- **plans 目录归档规范**：日期前缀命名（`YYYY-MM-DD-xxx.md`），design 文档与实施文档成对出现。
- **release_note 结构清晰**：顶层 release_note 有版本级别、Git 提交、变更摘要等完整元数据。
- **真相优先级声明**：多个模块 README 开篇即声明"公开头文件 > 实现 > examples > test > benchmark > Markdown"，文档定位明确。
- **无未闭合代码块**：全部模块文档的 ``` 围栏成对闭合，无语法错误。
- **无空文件/极小文件**：所有文档都有实质内容。
