# 01-架构设计

本页描述当前源码已经实现的结构，而不是理想化设计。

## 1. 目录分层

| 目录 | 职责 |
| --- | --- |
| `src/cpp/galay-etcd/base/` | 公共配置、错误类型、数据模型、etcd JSON 编解码、endpoint 解析 |
| `src/cpp/galay-etcd/sync/` | `EtcdClient`，基于阻塞 socket + 手写 HTTP/1.1 请求/响应处理 |
| `src/cpp/galay-etcd/async/` | `AsyncEtcdClient` 与 `AsyncEtcdClusterClient` 无锁池，基于 `galay::async::TcpSocket` 与 `galay::http::HttpSessionAwaitable` |
| `src/cpp/galay-etcd/cluster/` | 独立的 `EtcdClusterState` 策略状态与同步 `EtcdClusterClient` 无锁池 |
| `src/cpp/galay-etcd/module/` | `galay.etcd` C++ module 接口 |
| `examples/etcd/` | 公开可运行示例 |
| `test/etcd/` | 功能与解析验证程序 |
| `benchmark/etcd/` | 同步、异步与 cluster policy benchmark 入口 |

## 2. 共享基础层

基础层负责三件事：

1. 把 etcd 请求编码成 JSON
2. 把 etcd 响应解析回 `EtcdKeyValue` / `PipelineItemResult`
3. 统一 endpoint、`api_prefix`、错误和结果缓存语义

关键共享组件：

- `EtcdNetworkConfig`：`request_timeout`、`buffer_size`、`keepalive`
- `EtcdConfig`：在网络参数上补充 `endpoint` 与 `api_prefix`
- `EtcdProductionConfig`：多端点列表、policy、retry、健康检查参数
- `EtcdError` / `EtcdErrorType`
- `PipelineOp` / `PipelineItemResult`

## 3. Cluster 策略层

当前多端点能力由“独立策略状态”和“client 资源池”组成：

1. `EtcdClusterState` 仍可独立维护 endpoint 选择、health snapshot、retry/backoff 和 `EtcdClientStats`
2. `EtcdClusterClient` 为每个 endpoint 创建固定数量的 `EtcdClient`
3. `AsyncEtcdClusterClient` 为每个 endpoint 创建固定数量、绑定同一 `IOScheduler` 的 `AsyncEtcdClient`
4. 两个池都通过 moodycamel 无锁队列管理空闲 client，并通过 move-only RAII 租约保证独占使用
5. 池不代替调用方执行 connect、KV 请求、重试或健康检查

当前明确支持：

- `EtcdEndpointPolicy::FirstHealthy`
- `EtcdEndpointPolicy::RoundRobin`
- `connections_per_endpoint` 控制每个 endpoint 的 client 数量
- `tryAcquire()` 在池空时返回 `EtcdErrorType::PoolExhausted`
- `EtcdClientLease` / `AsyncEtcdClientLease` 析构时自动归还 client
- health snapshot 中的 `Unknown / Healthy / Unhealthy`
- `last_error`、`last_failure_time`、`last_success_time` 与 `last_probe_time`
- 针对网络错误和服务端错误的不同 retry decision

当前还没有实现：

- 实际 leader 探测；当前 `StickyLeader` 仅维护“最近一次成功端点”的 leader hint，健康时优先回到该端点，不等同于真实 leader status 感知
- pool 内部透明重试或请求转发
- 后台独立连接维护或 health probe 协程

## 4. 同步链路

`EtcdClient` 的同步路径是：

1. 构造时解析 endpoint、标准化 `api_prefix`
2. `connect()` 建立 TCP 连接
3. `put/get/del/grantLease/keepAliveOnce/pipeline` 发送 JSON POST
4. 解析 HTTP 状态码与响应体
5. 更新最近一次结果缓存

同步实现的重要事实：

- 使用阻塞 socket，但 `connect()` 会在配置了 `request_timeout` 时对建连应用超时
- `config.keepalive=true` 会影响 HTTP `Connection` 头与 TCP `SO_KEEPALIVE`
- `https://` endpoint 会在构造阶段被拒绝，因为当前没有 TLS 传输实现

## 5. 异步链路

`AsyncEtcdClient` 依赖 `IOScheduler*`。典型链路是：

1. 通过 `AsyncEtcdClientBuilder().scheduler(...).config(...).build()` 构造客户端
2. `co_await client.connect()`
3. `co_await client.put/get/del/...`
4. `co_await client.close()`

异步实现的重要事实：

- 没有 `IOScheduler*` 时，`connect()` 会返回 `EtcdErrorType::Internal`
- 请求阶段通过 `HttpSessionAwaitable` 处理 HTTP 往返
- `request_timeout` 用于 HTTP 请求 awaitable，不等于“异步 connect 超时”
- 同样不支持 `https://`
- `AsyncEtcdClusterClient` 只负责租借 `AsyncEtcdClient`；取得租约后仍由调用协程执行真实 HTTP I/O

## 6. `api_prefix` 与 endpoint 解析

源码中的两个独立概念：

- `endpoint`：主机、协议、端口
- `api_prefix`：默认 `/v3`

当前行为：

- `api_prefix` 会被标准化为“有前导斜杠、无尾部斜杠”
- endpoint 必须带 `http://` 或 `https://`
- 端口可省略；省略时 HTTP 默认 `80`、HTTPS 默认 `443`
- endpoint 中的路径部分不应作为 API 前缀使用；如需切换到非 `/v3` 前缀，请设置 `api_prefix`

## 7. Pipeline 语义

当前 public API 中的 “pipeline” 实际上是对 etcd `POST /v3/kv/txn` 的固定包装：

- `compare` 固定为空数组
- 所有操作都进入 `success`
- `failure` 固定为空数组

这意味着：

- 它是“一次 txn 请求”，不是客户端逐条循环发多个 HTTP 请求
- 当前库没有公开 compare 条件、failure 分支或 CAS/锁语义构造器
- 成功时 `EtcdPipelineResult` 里的结果顺序与输入操作顺序一致
- 失败时表现为整个 `std::expected` 出错，而不是“每项半成功半失败”的公开契约

## 8. 直接返回结果

当前公开 API 不再暴露 `last*()` 最近结果缓存，而是把结构化值直接放进返回值：

- `connect()` / `close()` / `put()` 返回 `EtcdBoolResult`
- `get()` 返回 `EtcdGetResult`
- `del()` 返回 `EtcdDeleteResult`
- `grantLease()` / `keepAliveOnce()` 返回 `EtcdLeaseGrantResult`
- `pipeline()` 返回 `EtcdPipelineResult`
- 异步 `co_await` 路径与同步路径保持同一语义
- cluster 池不包装这些操作；调用方从租约取得 client 后使用相同的直接返回结果语义

这意味着：

- 结果读取是“单次调用局部语义”，不会再依赖共享缓存
- 并发场景仍不建议共享同一个客户端实例，但不再有 `last*` 覆盖问题

## 9. 线程与并发边界

当前仓库没有把 `EtcdClient` 或 `AsyncEtcdClient` 标记为线程安全类型。推荐策略：

- 同步与异步单 client 都应保持串行使用
- 多调用方共享时优先共享 cluster pool，每次操作前 `tryAcquire()` 一个独占租约
- `AsyncEtcdClient` 不能跨越其绑定的 `IOScheduler` 使用
- `tryAcquire()` 不等待；池耗尽时由调用方决定重试、降级或返回错误

## 10. 公开边界之外

以下能力没有进入当前公开 API，文档中不应假设其存在：

- TLS / HTTPS 传输
- 用户名密码或 token 认证
- 除现有 async single-key watch 之外的 watch 生命周期恢复能力
- etcd 官方 Lock / Election / Cluster 管理 API
- compare/failure 事务构造器
- cluster pool 内部透明 failover / retry
