## 2. 命令行参数

`B1-EtcdKvBenchmark` 的参数顺序是：

1. `endpoint`
2. `threads`
3. `ops_per_thread`
4. `value_size`
5. `mode`：`put` 或 `mixed`

推荐命令：

```bash
./build-release/benchmark/cpp/etcd/benchmark_etcd_b1_kv http://127.0.0.1:2379 8 500 64 put
./build-release/benchmark/cpp/etcd/benchmark_etcd_b1_kv http://127.0.0.1:2379 8 500 64 mixed
```

`B2-AsyncEtcdKvBenchmark` 的参数顺序是：

1. `endpoint`
2. `workers`
3. `ops_per_worker`
4. `value_size`
5. `mode`：`put` 或 `mixed`
6. `io_schedulers`：可选，默认自动推导

推荐命令：

```bash
./build-release/benchmark/cpp/etcd/benchmark_etcd_b2_kv http://127.0.0.1:2379 8 500 64 put
./build-release/benchmark/cpp/etcd/benchmark_etcd_b2_kv http://127.0.0.1:2379 8 500 64 mixed
./build-release/benchmark/cpp/etcd/benchmark_etcd_b2_kv http://127.0.0.1:2379 32 500 64 put 8
```

环境变量：无。

## 3. Benchmark 实际统计项

`B1-EtcdKvBenchmark` 和 `B2-AsyncEtcdKvBenchmark` 当前都会输出：

- `Endpoint`
- `Mode`
- `Workers`
- `Ops/worker`
- `Value size`
- `IO schedulers`：仅 `B2`
- `Total ops`
- `Success`
- `Failure`
- `Duration`
- `Throughput`
- `Latency p50`
- `Latency p95`
- `Latency p99`
- `Latency max`

## 4. 运行前提与注意事项

- 需要一个可写的 etcd v3 HTTP endpoint
- benchmark 会写入 `/galay-etcd/bench/<timestamp>/...`
- `B2` 会写入 `/galay-etcd/bench/async/<timestamp>/...`
- benchmark 目标当前不负责清理所有写入数据
- `mixed` 模式是 `put` 后立即 `get`，不是只读压测
- `B2` 的结果依赖 `workers` 与 `io_schedulers` 组合，不同组合不应混写成同一组性能数字

## 7. Rust 对照 benchmark

仓库现在提供与 `B1/B2` 对齐的 Rust 基线：

- `benchmark/compare/rust/src/bin/rust_sync_etcd_bench.rs`
- `benchmark/compare/rust/src/bin/rust_async_etcd_bench.rs`
- `scripts/S2-Bench-Rust-Compare.sh`

Rust 基线选型：

- 同步路径：`reqwest::blocking`
- 异步路径：`reqwest` + `tokio`

统一对比入口：

```bash
GALAY_ETCD_ENDPOINT=http://127.0.0.1:2379 \
GALAY_ETCD_BENCH_WORKERS=4 \
GALAY_ETCD_BENCH_OPS_PER_WORKER=125 \
GALAY_ETCD_BENCH_VALUE_SIZE=32 \
GALAY_ETCD_BENCH_MODE=put \
GALAY_ETCD_BENCH_IO_SCHEDULERS=2 \
./scripts/S2-Bench-Rust-Compare.sh ./build
```

脚本会在两个场景下输出 C++ / Rust 的同参结果：

- `B1-EtcdKvBenchmark` vs `rust_sync_etcd_bench`
- `B2-AsyncEtcdKvBenchmark` vs `rust_async_etcd_bench`

输出摘要会包含：

- `start_time` / `end_time`（UTC）
- `throughput`
- `Latency p50/p95/p99`
- 资源指标仍需人工采集的提示

当前脚本默认值也与这组公开 compare workload 对齐：`workers=4`、`ops_per_worker=125`、`value_size=32`、`mode=put`、`io_schedulers=2`。

## 8. 对外发布规则

- 任何对外发布的 etcd benchmark 结论，都必须附上同机、同构建类型、同 workload 的 Rust 基线结果。
- 发布前必须先跑 `scripts/S2-Bench-Rust-Compare.sh`，并保存该脚本的原始 stdout。
- 如果某个数字没有对应 Rust 基线，只能标注为 `internal-only` 或历史样本，不能作为公开性能结论。

## 9. 重新发布性能数字的最低要求

如果你要把新的 benchmark 结果写回文档，至少同时记录：

- 执行日期
- 完整命令
- etcd 版本
- 机器规格（CPU / 内存 / OS）
- 编译器与构建类型
- 原始输出全文

没有这些元数据时，请不要把数字写成“当前性能”。

## 10. 结论

本轮只完成环境与依赖探测，没有产生可用于比较的性能数据。当前阻塞项有三个：

1. `../build-release` 是正确的 Release benchmark 配置，但 etcd benchmark 可执行文件尚未构建出来。
2. `127.0.0.1:2379` 没有 etcd 服务监听，无法执行 endpoint health、KV、Watch 或 Lease 测试。
3. 竞品 `etcd-cpp-apiv3` 在本机不可用，无法建立同机 C++ 竞品基线。

因此，KV 操作、Watch 延迟和 Lease 性能结论均为 `blocked`，不能写入吞吐或延迟数字。补跑时必须使用专用可写 etcd 实例、Release benchmark 二进制、完整原始输出和同 workload 竞品基线。
