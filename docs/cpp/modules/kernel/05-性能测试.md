## 当前已落地并已完成 fresh 回归确认的补充点

1. `B8` latency benchmark 增加 consumer-ready barrier
2. `MpscChannel` 增加 recv-side single-consumer prefetch
3. `MpscChannel` 的默认 batch size / prefetch limit 改为构造参数配置
4. `B6-Udp` 增加尾部 drain，降低 packet loss 抖动
5. `B1` latency 改为使用 `min(scheduler_count, 2)`，避免 burst 队列长度淹没调度开销
6. `B4-UdpServer` 将统计逻辑移出 scheduler 协程，避免阻塞事件循环并恢复 `B4/B5` 有效收发

## 解释 benchmark 时的注意点

- `B8-MpscChannel` 对 backlog、跨线程 wake 和 batch drain 边界很敏感
- `B14-SchedulerInjectedWakeup` 是 injected fast-path 的 canary
- `B10-Ringbuffer` 与 `B13-Sendfile` 对缓存状态、CPU 频率和单机环境非常敏感
- 网络 benchmark 需要明确 loopback / 容器 / 实机环境
- 文件 benchmark 需要明确缓存状态和文件大小

### 7. 结论与优化建议

结论：

1. 本轮只能确认当前 build 目录不可用于正式 kernel 性能压测，不能形成 Galay vs tokio/libuv/Boost.Asio 的性能优劣结论。
2. Galay kernel 的相关 benchmark 源码已存在，优先级可按 `B1/B14/B18` 调度延迟、`B8` MPSC channel、`B20` timer 与 C API `B24` sleep 精度补跑。
3. 竞品对比缺少可复用 harness；tokio 和 libuv 的库/工具链可作为后续基线基础，Boost.Asio 需先安装依赖。

优化建议：

1. 新增独立 `build-kernel-bench-release` 或 CMake preset，固定 `Release + GALAY_BUILD_BENCHMARKS=ON`，避免误用 Debug 构建。
2. 给 `b1`、`b8` 等 benchmark 增加命令行参数或配置文件，使 `RUNTIME_THREADS=4`、`TASKS_COUNT=100000`、`DURATION=30`、`WARM_UP=5` 能与计划严格一致。
3. 为 tokio/libuv/Boost.Asio 建立同目录竞品 harness，并输出统一 CSV 字段：`implementation,scenario,metric,value,unit,status,notes`。
4. timer/sleep 精度建议统一输出 p50/p95/p99/max 与 overshoot，而不是只记录平均值，便于和 tokio timer、libuv timer、Boost.Asio steady_timer 对齐。
