# WebSocket 性能测试

本文记录 `galay-ws` 的性能测试计划、当前构建与工具探测结果、可执行状态和补跑命令。当前会话没有产生吞吐、延迟或资源占用数值；凡是未运行的场景均明确标记为 `blocked` 或 `unavailable`，不填充推测数据。

## 1. 内部性能测试

当前仓库存在以下 WS benchmark 源码：

- `benchmark/cpp/ws/b5_ws_server_throughput.cc`: WS echo server benchmark。
- `benchmark/cpp/ws/b6_ws_client_throughput.cc`: WS client echo throughput / average RTT benchmark。
- `benchmark/cpp/ws/b7_wss_server_throughput.cc`: WSS server benchmark，需要 SSL 功能可用。
- `benchmark/cpp/ws/b8_wss_client_throughput.cc`: WSS client benchmark，需要 SSL 功能可用。
- `benchmark/cpp/ws/b9_protocol_boundaries.cc`: WS protocol boundary benchmark。
- `benchmark/c/ws/b1_message_throughput.c`: C API frame encode/decode throughput benchmark，需要 `GALAY_BUILD_C_API=ON`。
- `benchmark/c/ws/b2_frame_codec_pressure.c`: C API frame codec pressure benchmark，需要 `GALAY_BUILD_C_API=ON`。

## 2. 竞品对比测试

测试计划要求覆盖三类场景：

- Echo 延迟：单连接 ping-pong 往返延迟，计划工具为 `websocat`。
- 广播吞吐：1 对 N 广播性能，N 个客户端同时接收。
- 并发连接：最大并发 WebSocket 连接数。

统一计划参数：

| 参数 | 值 | 说明 |

| `WS_DURATION` | 30 s | 压测持续时间。 |
| `WS_SERVER_THREADS` | 4 | 服务端线程数。 |

## 3. 综合分析

当前无法给出 Galay 与 websocketd、gorilla/websocket、node/ws 的性能胜负结论。原因不是测试失败，而是当前构建和工具链不满足计划前提：

- Galay 当前构建不是 Release，且 benchmark 没有开启。
- 当前 build 图没有可执行 WS benchmark target。
- 计划要求的 `websocat` 和 `websocketd` 均不可用。
- gorilla/websocket 依赖缓存存在，但没有本轮可直接运行的同口径 benchmark harness。
- node/ws 包不可用，也没有本轮可直接运行的同口径 benchmark harness。

## 4. 性能瓶颈分析

本轮没有运行端到端 WS benchmark，因此不做热点、CPU、内存或 IO 瓶颈断言。

下一轮可在可运行数据产生后补充：

- echo 热路径的 RTT 分布和吞吐。
- server-side CPU 热点，例如 WS frame 编解码、mask/unmask、socket read/write、调度器唤醒。
- 大连接数下的内存占用、fd 使用、连接关闭成本和失败原因。
- 广播场景下的消息复制、发送队列堆积和慢客户端影响。

## 5. 优化建议

短期：

1. 建立独立 Release benchmark 构建目录，并开启 `GALAY_BUILD_BENCHMARKS=ON`。
2. 安装或提供 `websocat`、`websocketd`，并固定版本。
3. 为 gorilla/websocket 和 node/ws 增加同口径 server/client harness，或指定已有脚本路径。
4. 扩展 WS client benchmark 输出 p50/p95/p99，避免只依赖 min/avg/max。

中期：

1. 增加广播 benchmark，记录每客户端接收完整性和慢客户端行为。
2. 增加并发连接阶梯 benchmark，输出最大稳定连接数和失败原因分布。
3. 将每轮结果归档为 CSV，并保留启动命令、参数、版本和环境。
