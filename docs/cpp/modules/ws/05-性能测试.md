## 3. 综合分析

当前无法给出 Galay 与 websocketd、gorilla/websocket、node/ws 的性能胜负结论。原因不是测试失败，而是当前构建和工具链不满足计划前提：

- Galay 当前构建不是 Release，且 benchmark 没有开启。
- 当前 build 图没有可执行 WS benchmark target。
- 计划要求的 `websocat` 和 `websocketd` 均不可用。
- gorilla/websocket 依赖缓存存在，但没有本轮可直接运行的同口径 benchmark harness。
- node/ws 包不可用，也没有本轮可直接运行的同口径 benchmark harness。

## 4. 性能瓶颈分析

本轮没有运行端到端 WS benchmark，因此不做热点、CPU、内存或 IO 瓶颈断言。

下一轮可在可运行数据产生后补充：

- echo 热路径的 RTT 分布和吞吐。
- server-side CPU 热点，例如 WS frame 编解码、mask/unmask、socket read/write、调度器唤醒。
- 大连接数下的内存占用、fd 使用、连接关闭成本和失败原因。
- 广播场景下的消息复制、发送队列堆积和慢客户端影响。

## 5. 优化建议

短期：

1. 建立独立 Release benchmark 构建目录，并开启 `GALAY_BUILD_BENCHMARKS=ON`。
2. 安装或提供 `websocat`、`websocketd`，并固定版本。
3. 为 gorilla/websocket 和 node/ws 增加同口径 server/client harness，或指定已有脚本路径。
4. 扩展 WS client benchmark 输出 p50/p95/p99，避免只依赖 min/avg/max。

中期：

1. 增加广播 benchmark，记录每客户端接收完整性和慢客户端行为。
2. 增加并发连接阶梯 benchmark，输出最大稳定连接数和失败原因分布。
3. 将每轮结果归档为 CSV，并保留启动命令、参数、版本和环境。

### 受控重跑结果

**实测数据**: [ws_same_class_compare_controlled_2026-07-04.csv](./benchmark_data/ws_same_class_compare_controlled_2026-07-04.csv)  
**资源采样**: [ws_same_class_resource_controlled_2026-07-04.csv](./benchmark_data/ws_same_class_resource_controlled_2026-07-04.csv)

受控重跑采用固定 gorilla 二进制 `/tmp/galay_ws_gorilla_echo_20260704`，不再使用 `go run`；每轮开始前清理 `18080-18099` 残留监听进程，Galay benchmark 重新确认 `build-release` 为 `Release` 且 `CMAKE_CXX_FLAGS_RELEASE=-O3 -DNDEBUG`。

| implementation | avg msg/s | avg RTT | successful | failed | server OS threads peak | server running threads peak | 备注 |
|---|---:|---:|---:|---:|---:|---:|---|
| galay-ws | 151,033 | 6.602 ms | 1000/1000 | 0 | 6 | 1 | 3 轮：153,818 / 149,296 / 149,985 msg/s |
| gorilla/websocket | 172,260 | 5.784 ms | 1000/1000 | 0 | 15 | 5 | 3 轮：171,879 / 171,418 / 173,483 msg/s |

本口径下 gorilla/websocket 的平均吞吐约为 Galay 的 `1.140x`，平均 RTT 约低 `12.4%`。但该结果不是“相同 server 资源下 Go 更强”的结论：负载下 gorilla server 的 OS 线程数峰值为 `13-15`，running thread 峰值为 `4-5`；Galay server 固定为 `6` 个 OS 线程，running thread 峰值为 `1-2`。

补充资源采样轮使用 `ps` 记录 CPU/RSS，原因是本机 macOS `top -pid` 对过滤进程的 `%CPU` 字段一直显示 `0.0`，不能作为 CPU 依据：

| implementation | msg/s | avg RTT | server OS threads peak | server running threads peak | server CPU peak | server RSS peak |
|---|---:|---:|---:|---:|---:|---:|
| galay-ws | 153,060 | 6.514 ms | 6 | 2 | 100.4% | 75,776 KB |
| gorilla/websocket | 172,158 | 5.793 ms | 14 | 4 | 290.0% | 52,912 KB |

因此当前更严谨的解释是：gorilla/websocket 在该 echo fixture 下吞吐更高，但它的 Go runtime 在负载下会创建/唤醒更多 OS 线程，`GOMAXPROCS=4` 只限制同时运行 Go 代码的 P 数量，不等同于 server 总线程数上限。若要证明 Galay 与 gorilla 在“等 CPU/等 running threads”下的差异，需要继续做 CPU 亲和性或进程级 CPU 限额实验，或调整 Galay server 的调度器配置让 running threads 达到同一资源 envelope。

本结论仅覆盖本机 h1 WebSocket echo、1000 连接、1KB payload、同一个 Galay client 的 ping-pong workload，不外推到广播、TLS/WSS、最大连接数或浏览器客户端。

## 2026-07-14 Linux epoll 与 libwebsockets 固定口径补测

为确认旧的 `-17.53%` 是否来自 5 秒短测噪声，本轮使用同一个 Galay epoll 客户端，对 Galay epoll 与
libwebsockets 4.5.8 做轮换顺序补测。负载为 32 连接、1024B ping-pong，服务端均为 1 个 IO thread；每轮先
warmup 5 秒，再正式运行 30 秒，服务端/客户端 CPU 分别固定为 `0-1` / `2-3`。

| 实现 | run1 | run2 | run3 | 中位数 | CV | 中位平均 RTT | 连接结果 |
|---|---:|---:|---:|---:|---:|---:|---:|
| Galay epoll | 56,173.2 | 52,332.1 | 52,841.4 | **52,841.4 msg/s** | 3.167% | 0.604824 ms | 32/32，0 失败 |
| libwebsockets 4.5.8 | 75,840.1 | 65,879.4 | 67,929.2 | **67,929.2 msg/s** | 6.146% | 0.470321 ms | 32/32，0 失败 |

Galay epoll 的中位吞吐比 libwebsockets 低 **22.21%**，中位平均 RTT 高 **28.60%**。libwebsockets 的 CV
略高于 5%，因此精确差值仍需保守解释；但三轮范围没有重叠，其最低轮 `65,879.4 msg/s` 仍比 Galay 最高轮
`56,173.2 msg/s` 高 **17.28%**，足以确认 epoll 差距不是短测噪声。

结合前一节 profile，本轮结论不是实施 `WsMessageReadState` 对象池，而是把下一优化杠杆收敛到网络唤醒、
`readv` / `sendmsg`、`epoll_ctl` 与 sequence 注册切换。该方向会改变 reactor 契约，实施前需要针对重复注册、
读写兴趣切换、连接关闭和并发唤醒补边界测试与压力测试。

完整样本：[ws_epoll_competitor_2026-07-14.csv](./benchmark_data/ws_epoll_competitor_2026-07-14.csv)。

## 2026-07-14 epoll 持久 READ 注册优化

实现前先用同源代码构建独立 baseline，并只回退本次 epoll 改动。复核发现计划最初把 WS echo 热路径描述成简单
`ReadvAwaitable`，但当前 `WsConnImpl::echoOnce` 实际使用 `StateMachineAwaitable`/sequence。只优化简单 readv 时，
`epoll_ctl` 调用量与 baseline 几乎相同，吞吐也没有提升；最终实现因此同时覆盖普通 `RECV/READV` 和 sequence
的 READ 兴趣，在 controller `remove`/close 前保持 `EPOLLIN | EPOLLET` 常驻，WRITE 仍按需注册。

固定口径仍为 5 秒 warmup + 30 秒、32 连接、1024B ping-pong、服务端 1 个 IO thread，服务端/客户端 CPU
分别固定 `0-1` / `2-3`：

| 实现 | run1 | run2 | run3 | 中位数 | CV | 中位平均 RTT | 失败 |
|---|---:|---:|---:|---:|---:|---:|---:|
| 同源 baseline | 52,251.5 | 52,802.9 | 57,041.3 | **52,802.9 msg/s** | 3.960% | 0.605333 ms | 0 |
| 持久 READ | 54,686.8 | 55,095.4 | 54,842.3 | **54,842.3 msg/s** | 0.307% | 0.582805 ms | 0 |

中位吞吐提升 **3.86%**，中位平均 RTT 下降 **3.72%**。相对前一节 libwebsockets 的历史固定口径中位数，
吞吐差距约从 `-22.21%` 收窄到 `-19.27%`；该比较不是本轮交替顺序重测，只用于观察差距量级。

10 秒 `strace -f -c` 对照使用相同的 32 连接/1024B/绑核配置。由于 strace 本身显著降低吞吐，调用量只用于
前后相对比较：

| 实现 | client msg/s | `readv` | `sendmsg` | `epoll_wait` | `epoll_ctl` |
|---|---:|---:|---:|---:|---:|
| baseline | 6,632.3 | 69,896 | 66,387 | 311 | 7,022 |
| 持久 READ | 6,866.7 | 71,848 | 68,731 | 301 | **36** |

`epoll_ctl` 调用下降 **99.49%**，剩余 36 次为连接建立/清理等非稳态注册。最终 Linux Release 定向回归
18/18 通过；ASan+UBSan 下持久读、sequence、双向 sequence 和 WS echo 4/4 通过。
