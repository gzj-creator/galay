## 构建 benchmark

要生成 benchmark 二进制，必须启用 `GALAY_BUILD_BENCHMARKS`：

```bash
rtk cmake -S . -B build-release \
  -DCMAKE_BUILD_TYPE=Release \
  -DGALAY_BUILD_BENCHMARKS=ON
rtk cmake --build build-release --target benchmark_mysql_sync_query_pressure benchmark_mysql_async_query_pressure benchmark_mysql_async_pool_pressure benchmark_mysql_async_pool_lease_pressure --parallel
```

## 发布要求

- 发布任何 benchmark 结论前，必须跑 `scripts/S2-Bench-Rust-Compare.sh`，确保 C++ 与 Rust 在同一组参数下比对，本脚本会在摘要里打印 `start_time`/`end_time`（UTC）、吞吐和 p50/p95/p99 延迟，并附加资源指标仍需人工采集的说明。
- 将脚本原始 stdout 通过 `tee` 或重定向保存到带有时间戳的文件（例如 `benchmark-results/2026-04-12-S2-Bench-Rust-Compare.txt`），文件内就包含了命令、环境变量、开始与结束时间。该文件加上 start_time、commit 及测试日期便是公开数字的主要证据链。
- 资源占用（CPU、内存、网络等）目前还没有自动化采集，请在执行 benchmark 时用 `top`、`ps`、平台面板等手动截取快照，并把这些补充记录与 stdout/命令一起归档。

## 如何解释 benchmark 模式

- `normal`：按默认单条语句流程执行
- `batch`：使用批量命令模式
- `pipeline`：使用 pipeline 模式发送多条语句

具体语义仍以 `benchmark/common/config.h` 与相应 benchmark 源码为准。

## 2026-07-13 全模块计划执行结果

本节为旧探测记录之后的新鲜 Release 实测。Galay 为 `v4.1.0-2-gadd1629-dirty`，
构建参数为 `Release`、`-O3 -DNDEBUG`、`GALAY_BUILD_BENCHMARKS=ON`。

新增 `benchmark_mysql_libmysqlclient_query_pressure`，使用 mysqlclient pkg-config 版本 24
（MySQL 9.3.0），并与 Galay `benchmark_mysql_sync_query_pressure` 复用相同 CLI/环境变量口径。
统一脚本为 `scripts/mysql/300_mysql_compare_libmysqlclient.sh`。

正式 workload：`SELECT 1`、8 clients、每 client 1000 queries、100 次 warmup、三轮取中位数。

| 实现 | QPS 中位数 | p50 | p95 | p99 | 失败 |
|---|---:|---:|---:|---:|---:|
| Galay sync client | 131,946 | 0.053917 ms | 0.072250 ms | 0.095708 ms | 0 |
| libmysqlclient | 96,659.4 | 0.054541 ms | 0.075917 ms | 0.105417 ms | 0 |

在此只读小查询口径下，Galay QPS 中位数比 libmysqlclient 高 36.50%，p99 低约 9.21%。
三轮所有 48,000 次查询均成功。该结论只适用于 loopback、持久连接、`SELECT 1`，不能外推到结果集解码或写事务。

mysqlslap 9.3.0 也以 8 clients、总计 8000 queries 各跑三轮，中位数约 117,647 QPS。
mysqlslap 不提供与两个客户端 harness 相同的逐请求 p50/p99，且其线程/计时实现不同，因此只列官方参考，不进入上表百分比排名。
