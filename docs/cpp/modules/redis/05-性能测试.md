## 基础命令测试

计划场景要求覆盖 GET/SET/INCR/HGET/LPUSH。本次短测条件不满足，以下是补跑命令。

启动 Redis：

```bash
rtk redis-server --port 6379 --save "" --appendonly no
```

官方 GET/SET 基线：

```bash
rtk redis-benchmark -h 127.0.0.1 -p 6379 \
  -c 50 -n 100000 -d 64 -P 1 -t get,set
```

Galay 单连接 GET/SET 短测：

```bash
rtk ./build-release/benchmark/cpp/redis/benchmark_redis_async_client_throughput \
  -h 127.0.0.1 -p 6379 \
  -c 10 -n 10000 -m normal \
  --timeout-ms -1 --buffer-size 32768 -q
```

当前 Galay `normal` benchmark 会为每个 operation 执行一组 SET/GET，并输出 `Ops/sec`、`Request latency p50 (us)`、`Request latency p99 (us)`、成功/错误/超时计数。它不暴露 `-d/--data-size` 参数，因此与 `redis-benchmark -d 64` 对比时需要在报告中注明负载生成方式不同。

## Pipeline 测试

计划场景要求 pipeline 深度 `10/50/100`，并与非 pipeline 对比。本次未运行，补跑命令如下。

官方 pipeline 基线：

```bash
rtk redis-benchmark -h 127.0.0.1 -p 6379 \
  -c 50 -n 100000 -d 64 -P 100 -t get,set
```

Galay pipeline：

```bash
rtk ./build-release/benchmark/cpp/redis/benchmark_redis_async_client_throughput \
  -h 127.0.0.1 -p 6379 \
  -c 10 -n 50000 -m pipeline -b 100 \
  --timeout-ms -1 --buffer-size 32768 -q
```

Galay `pipeline` 通过 `RedisCommandBuilder::encoded()` 和 `RedisClient::batchBorrowed(...)` 发送预编码批次；它代表 borrowed fast path 的吞吐，不应直接泛化为所有 owning API 调用路径。

## 注意事项

- `--track-alloc` 会改变开销，不适合与未开启统计的结果混合比较。
- `normal` / `pipeline` 当前走 borrowed fast path；跨实现对比时必须注明调用路径。
- pipeline 深度、clients、operations、timeout、buffer size 必须在 Galay 与竞品之间保持一致。
- 计划要求至少 3 次测试取中位数；本次未满足可运行条件，因此没有图表归档。

## 2026-07-13 全模块计划执行结果

本节覆盖旧探测记录之后的新鲜 Release 实测。执行环境见
[测试机器配置](../../../machine_config.md)，Galay 为 `v4.1.0-2-gadd1629-dirty`，
构建参数为 `Release`、`-O3 -DNDEBUG`、`GALAY_BUILD_BENCHMARKS=ON`。

新增同机竞品目标：

- `benchmark_redis_hiredis_client_throughput`：hiredis 1.3.0，同步多连接客户端
- `scripts/redis/300_redis_compare_hiredis.sh`：启动隔离 Redis 8.0.1，执行三轮、零错误门禁并归档原始输出

正式参数：10 clients、每 client 5000 operations、timeout 关闭；pipeline 深度为 10/50/100。
`normal` 两侧均执行预构造 key/value 的 SET+GET；`pipeline` 两侧均执行批量 SET。

### 三轮中位数

| 场景 | Galay Ops/s | hiredis Ops/s | Galay 相对值 | Galay p50 | hiredis p50 | Galay p99 | hiredis p99 |
|---|---:|---:|---:|---:|---:|---:|---:|
| normal SET+GET | 172,101 | 167,067 | +3.01% | 56 us | 55 us | 98 us | 101 us |
| pipeline 10 | 992,490 | 1,031,339 | -3.77% | 84 us/批 | 87 us/批 | 120 us/批 | 193 us/批 |
| pipeline 50 | 2,108,473 | 1,365,785 | +54.38% | 206 us/批 | 324 us/批 | 366 us/批 | 856 us/批 |
| pipeline 100 | 1,817,617 | 1,369,409 | +32.73% | 372 us/批 | 665 us/批 | 481 us/批 | 1,504 us/批 |

全部 24 个 Galay/hiredis 样本均为 `Error=0`、`Timeout=0`。高吞吐 pipeline 重跑前，
Galay B1 的整数毫秒计时会产生量化误差；本轮已改用浮点毫秒并重新生成全部正式数据。

### 连接池自对照

连接池使用 10 workers、固定 10 connections，每 worker 5000 次 SET+GET。三轮中位数为
60,532 Ops/s；相同 operations 口径的 dedicated-connection `normal` 为 172,101 Ops/s。
该结果只说明当前 B2 连接池路径在此配置下的 lease/调度开销，不外推到不同 pool size 或竞争强度。

### 官方 redis-benchmark 参考

`redis-benchmark 8.0.1` 的 SET/GET/INCR、pipeline 1/10/50/100 已各跑三轮，原始 CSV 全部有效。
它使用官方固定命令生成器，而 Galay/hiredis 对照使用成对 SET+GET 或批量 SET，因此标记为
`official-reference-only`，不与上表做百分比排名。
