# 01-架构设计

本页按当前仓库真实目录和公开头文件描述系统结构，不再引用旧的 `client.get()` / `pipeline()` 风格 async API。

## 单连接 async 执行链路

当前主执行链路如下：

1. 用 `RedisCommandBuilder` 生成 `RedisEncodedCommand` 或 `std::span<const RedisCommandView>`
2. `RedisClient::command(...)` / `RedisClient::batch(...)` 发送 RESP 数据
3. `protocol::RespParser` 解析服务器返回
4. 调用侧通过 `RedisValue` 做类型判断与取值

对应源码：

- 命令构建：`galay-redis/protoc/builder.h`
- async 客户端：`galay-redis/async/redis_client.h`
- RESP 解析：`galay-redis/protoc/redis_protocol.h`
- 返回值：`galay-redis/base/redis_value.h`

## 拓扑层

### 主从 / Sentinel

`RedisMasterSlaveClient` 负责：

- `connectMaster(...)`：显式接入主节点
- `addReplica(...)`：注册副本连接
- `addSentinel(...)` / `refreshFromSentinel()`：从 Sentinel 拉取最新地址
- `execute(cmd, args, prefer_read, auto_retry)`：按读写偏好选择连接
- `batch(commands, prefer_read)`：在主节点或副本上批量发送

真实验证来源：

- API 形状与基本行为：`test/t11_pubsub.cc`
- 并发刷新去重：`test/t12_topology.cc`
- 真实 failover：`test/t13_cluster.cc`

### Cluster

`RedisClusterClient` 负责：

- `addNode(...)`：添加 seed 节点
- `refreshSlots()`：刷新槽位映射
- `keySlot(key)`：计算 key slot
- `execute(cmd, args, routing_key, auto_retry)`：按 `routing_key` 路由
- `batch(commands, routing_key)`：把一批命令打到同一路由节点

真实验证来源：

- key slot 与基本路由：`test/t11_pubsub.cc`
- MOVED / ASK / slot refresh：`test/t13_cluster.cc`

## 连接池层

`RedisConnectionPool` 基于 `RedisClient` 提供：

- 初始化与预热：`initialize()`、`warmup()`
- 获取/归还：`acquire()`、`release(...)`
- 健康检查与清理：`triggerHealthCheck()`、`triggerIdleCleanup()`
- 统计：`getStats()`

验证来源：

- 功能与 RAII：`test/t4_pool.cc`
- 压测：`benchmark/b2_pool.cc`

## 生命周期与并发边界

### 调度器绑定

`RedisClientBuilder`、`RedisMasterSlaveClientBuilder`、`RedisClusterClientBuilder` 都要求显式注入 `IOScheduler*`。示例和测试统一通过 `Runtime::getNextIOScheduler()` 取调度器。

### 移动语义限制

`RedisClient` 头文件明确写了两条限制：

- 不要在操作进行中移动 `RedisClient`
- 确保所有 awaitable 已经失效后再移动

因此文档不再把 `RedisClient` 描述为“可随意移动的轻量句柄”。

### Pub/Sub 读取路径

Pub/Sub 订阅后，真正读取消息使用的是：

```cpp
co_await subscriber.receive(1)
```

而不是某个专门的 `subscribeMessage()` 辅助 API。真实示例见 `examples/include/e3_pubsub.cc` 和 `test/t11_pubsub.cc`。

### 拓扑刷新 single-flight

`refreshFromSentinel()` 和 `refreshSlots()` 的并发去重行为由 `test/t12_topology.cc` 覆盖：同一波突发请求只会触发一次后台刷新。
