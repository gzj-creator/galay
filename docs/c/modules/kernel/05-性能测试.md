# 05-性能测试

## 当前有效口径（2026-06-25）

- 当前 C Kernel benchmark 只记录真实存在的 C target、源码文件和本地 fresh 输出。
- 当前网络吞吐主口径只包含真实双进程 client/server 互压：TCP echo 与 UDP echo。
- `benchmark_c_kernel_udp_socket_throughput` 是单进程 loopback datagram 压测，不作为 client/server 网络吞吐主结论。
- `benchmark_c_kernel_timeout_api_pressure` 和 timeout smoke target 是 API 压测/语义验证入口，不和 TCP/UDP QPS 或 MB/s 横向比较。
- TCP workload：server `2` 个 IO scheduler，client `1` 个 IO scheduler、`32` 条连接、`1024B request -> 1024B response`，持续 `5s`。
- UDP workload：server `2` 个 IO scheduler，client `1` 个 IO scheduler、`100` 个 client、每 client `1000` 条 `256B` datagram，持续 `5s`。
- 当前结果只代表本机 macOS kqueue loopback C API callback 路径，不代表跨机器吞吐或长稳压测。

## 解释 benchmark 时的注意点

- TCP/UDP throughput benchmark 都需要独立 server/client 两个进程；复现时 server 和 client 的端口必须一致。
- `QPS` 是完整 TCP request/response round-trip 次数，不是单向 `recv` 或 `send` 次数。
- TCP `throughput MB/s` 按 request payload 与 response payload 的总字节数计算，即 `requests * payload_bytes * 2 / elapsed_seconds / 1024 / 1024`。
- UDP `loss_percent` 按 client 收到的 echo 数量计算：`1 - received / sent`；server 侧收到并回发的数据包数用于辅助判断丢包发生在 client->server 还是 server->client 方向。
- C TCP server 使用 `galay_kernel_tcp_socket_accept_loop`，session 内使用 one-shot `recv/send`；C TCP client 使用 `recv_loop` 常驻接收并链式提交下一次 send。
- C UDP server 使用单 socket 的 one-shot `recvfrom -> sendto -> recvfrom` callback chain，避免同一 buffer 在 recv 和 send 之间并发复用。
- C UDP client 使用多个独立 UDP socket，每个 client 串行执行 `sendto -> recvfrom_timeout`，UDP 超时只计入丢包，不计入 API error。
- `benchmark_c_kernel_timeout_api_pressure` 是单进程、单 runtime、混合 TCP/UDP/FileWatcher/AsyncFile 的 timeout API 压力入口，不是网络吞吐 benchmark。
- `benchmark_c_kernel_aio_file_batch` 只在 `USE_EPOLL` 后端执行真实 AIO 批量提交；macOS/kqueue 或非 epoll 构建会输出 `operation unsupported` 并以成功状态退出。
- C++ `B11/B12` TCP IOV 当前没有 C ABI 对应 target；C 侧未暴露 `readv/writev` API，因此不在 C benchmark 清单中伪造对应项。
- 结果会受本机调度、CPU 频率、loopback 栈和 kqueue 唤醒开销影响；对比性能时应使用同一机器、同一构建参数、同一 target 串行重跑。
