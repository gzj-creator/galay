## 推荐复现命令

配置构建：

```bash
cmake -S . -B build/codex-release-benchmark \
  -DCMAKE_BUILD_TYPE=Release \
  -DGALAY_BUILD_C_API=ON \
  -DGALAY_BUILD_EXAMPLES=ON \
  -DGALAY_BUILD_BENCHMARKS=ON \
  -DBUILD_TESTING=ON \
  -DGALAY_ENABLE_CPP23_MODULES=OFF \
  -DGALAY_DISABLE_IOURING=ON
```

构建 C 网络 benchmark：

```bash
cmake --build build/codex-release-benchmark \
  --target benchmark_c_kernel_tcp_socket_server_throughput benchmark_c_kernel_tcp_socket_client_throughput
```

运行 C 网络 benchmark：

```bash
./build/codex-release-benchmark/benchmark/c/kernel/benchmark_c_kernel_tcp_socket_server_throughput 19081 2
./build/codex-release-benchmark/benchmark/c/kernel/benchmark_c_kernel_tcp_socket_client_throughput \
  -h 127.0.0.1 -p 19081 -c 32 -s 1024 -d 5 --io-schedulers 1
```

C++ 同参数本机对照：

```bash
cmake --build build/codex-release-benchmark \
```

## 解释 benchmark 时的注意点

- C 网络 benchmark 使用独立 server/client 两个进程；server 固定 `2` 个 IO scheduler，client 固定 `1` 个 IO scheduler、`32` 条连接、`5s` 时长和 `1024B` payload
- C server 使用 `galay_kernel_tcp_socket_accept_loop` 降低 accept 重复提交；session 内仍使用 one-shot `recv/send`，避免常驻 `recv_loop` 复用同一 buffer 覆盖正在发送的数据
- C client 使用 `galay_kernel_tcp_socket_recv_loop` 常驻接收，并在完整 response 到达后链式提交下一次 one-shot `send`
- loop callback 返回 `0` 表示继续下一轮，返回非 `0` 表示当前回调后退出 loop；压测停止不再依赖关闭对端触发错误退出
- C++ 对照使用相同连接数、payload、时长和 server/client scheduler 数
- `QPS` 是完整 request/response round-trip 次数，不是单向 `recv` 或 `send` 次数
- `throughput MB/s` 按 request payload 与 response payload 的总字节数计算，即 `requests * payload_bytes * 2 / elapsed_seconds / 1024 / 1024`
- C server 和 C client 都走 C `TcpSocket` 的 callback API；server 负责 `accept` / `recv` / `send` / `close`，client 负责 `connect` / `send` / `recv` / `close`
- `benchmark_c_kernel_tcp_socket_lifecycle` 每轮都会创建一个 POSIX client fd，用 C API 在 listener 上异步 `accept`，随后异步 `close` server-side socket
- lifecycle `ops_per_sec` 是完整生命周期次数，不是 request/response QPS
- 端口由系统分配，两个 benchmark 都不依赖固定端口
- 结果会受本机调度、CPU 频率、loopback 栈和 kqueue 唤醒开销影响；对比性能时应使用同一机器、同一构建参数、同一 target 串行重跑
