# CLAUDE.md

行为准则,用于减少 LLM 常见编码错误。可根据项目特定说明合并使用。

**权衡:** 这些准则倾向于谨慎而非速度。对于琐碎任务,可自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。主动呈现权衡。**

实现之前:
- 明确陈述你的假设。如不确定,先询问。
- 如果存在多种解释,呈现它们 - 不要默默选择。
- 如果存在更简单的方案,要指出来。适时反驳。
- 如果有不明确之处,停下来。指出困惑所在。提问。

## 2. 简洁优先

**用最少的代码解决问题。不做任何投机性工作。**

- 不添加超出需求的特性。
- 不为一次性代码做抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行而本可以 50 行完成,重写它。

问问自己:"资深工程师会说这过于复杂吗?"如果是,简化它。

## 3. 外科手术式修改

**只动必须动的地方。只清理自己造成的混乱。**

编辑现有代码时:
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格,即使你会采用不同的写法。
- 如果发现不相关的死代码,提一下 - 不要删除它。

当你的修改产生孤立内容时:
- 移除因你的修改而变得未使用的 import/变量/函数。
- 不要移除之前就存在的死代码,除非被要求。

检验标准:每一行改动都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标:
- "添加校验" → "为无效输入编写测试,然后让它们通过"
- "修复 bug" → "编写一个能复现该 bug 的测试,然后让它通过"
- "重构 X" → "确保前后测试都通过"

对于多步骤任务,陈述一个简要计划:
```
1. [步骤] → 验证:[检查项]
2. [步骤] → 验证:[检查项]
3. [步骤] → 验证:[检查项]
```

强成功标准让你能独立循环。弱标准("让它能跑")需要不断澄清。

## 5. 错误传播与异常禁用

**禁止使用异常隐藏错误。错误只能通过返回值显式传播。**

- C++ 生产代码禁止新增 `throw`、`try`、`catch`、`catch (...)` 或用异常做正常控制流。
- C++ 可恢复错误必须使用 `std::expected<T, E>` 或 `std::expected<void, E>` 返回,错误类型要能表达具体失败原因。
- 如果必须调用可能抛异常的第三方/遗留边界,必须先重新设计边界或把该风险隔离在已有边界内;不得新增吞异常兜底逻辑。
- C API 必须通过自定义结果结构体或明确错误码返回失败,不得依赖异常、全局隐式状态或只靠 out 参数表达失败。
- 每个公开 C 错误码枚举都必须提供对应的错误字符串获取函数,例如 `*_get_error(...)`,并覆盖所有枚举值。
- 跨 C/C++ 边界时,底层 errno、系统错误、C++ typed error 必须立即转换为公开错误码或 `std::expected` 的 error,不得丢失错误原因。

## 6. 函数返回值必须处理

**禁止丢弃函数返回值。每个函数调用的返回值都必须显式检查、传播或存储。**

- 不得使用 void-cast、裸调用或注释说明来忽略带返回值函数的结果。
- 清理/回滚路径也必须处理返回值；如果不能改变主错误码,至少要把清理失败合并到已有返回结构或转换为可观测的错误码。
- C++ 代码应通过 `std::expected`、布尔/枚举状态或结构化结果显式传播失败。
- C 代码应通过结果结构体、错误码或调用方可检查的状态字段显式传播失败。
- 只有 `void` 返回函数、无返回值宏、或仅用于标记未使用参数的 `(void)param` 可以不产生返回值处理。

---

**这些准则生效的表现:** diff 中不必要的改动更少,因过度复杂而重写的情况更少,澄清问题出现在实现之前而非犯错之后。
